## Context

ECE applications are currently built ad-hoc. The sandbox (`scripts/build-sandbox.sh`) manually assembles HTML, JS, WASM, and .ecec files. The WASM test suite concatenates .scm files into one blob before compiling. There is no general-purpose way to compile a multi-file ECE project and package it for distribution.

The `.ecec` format currently stores one compilation space per file (one header + one instruction list). Each .scm file compiled with `compile-file` produces one .ecec. The bootstrap loads 6 separate .ecec files in sequence. Cross-space calls work via global environment lookup.

## Goals / Non-Goals

**Goals:**
- Compile multiple .scm files into a single .ecec bundle preserving per-file spaces
- Provide `ece-build` as an SDK entry point for users working in their own repos
- Package web apps (HTML+WASM) and CL apps with one command
- Produce a bundle format suitable as input to future compile-to-host codegen

**Non-Goals:**
- Automatic `(load ...)` dependency tracing — users specify file order explicitly
- Watch mode, incremental builds, or caching
- Compile-to-host integration (next phase, consumes the bundle this produces)
- Package manager or dependency resolution for third-party libraries

## Decisions

### 1. Multi-space bundle = concatenated single-space .ecec sections

**Choice:** A bundle is N copies of `(ecec-header ...) (instruction-list ...)` concatenated in one file. Each section is exactly the output of `compile-file` for one .scm.

**Why not a wrapper format?** A wrapper (e.g., `(ecec-bundle (section ...) (section ...))`) would require a new parser. Concatenation means the existing `load-compiled` / `load-ecec-file` logic reads one section at a time — we just loop until EOF. Minimal code change, fully backward compatible.

**Why not merge into one space?** Separate spaces preserve per-file source-maps, keep the label namespaces isolated, and match the semantics of `(load ...)`. The bundle becomes the universal IR for compile-to-host — the codegen walks spaces individually.

### 2. `compile-system` calls `compile-file` internally, concatenates outputs

**Choice:** `compile-system` takes a list of .scm filenames and an output bundle path. For each file, it calls `compile-file` (which writes a temporary single-space .ecec), then concatenates all outputs into the bundle file. Alternatively, it can write headers + instructions directly without intermediate files.

**Implementation:** Write directly to the output port — for each .scm file, compile all forms, extract source-map, write ecec-header + instructions. This avoids temporary files and is a straightforward extension of `compile-file`.

```scheme
(compile-system '("src/utils.scm" "src/world.scm" "src/main.scm")
                "build/app.ecec")
```

### 3. `load-bundle` reads sections until EOF

**Choice:** `load-bundle` opens the bundle file and calls `load-compiled`'s inner logic in a loop until it hits EOF. Each section creates a new space, registers its source-map, and executes.

**Backward compatibility:** `load-compiled` continues to work for single-space files. `load-bundle` works for both single and multi-space files (a single-space file is just a 1-section bundle).

### 4. `ece-build` is a shell script in `bin/`

**Choice:** `bin/ece-build` is a POSIX shell script. It determines `ECE_HOME` relative to its own location (`$(dirname $0)/..`), boots ECE via `qlot exec sbcl`, calls `compile-system`, then copies runtime files and generates target-specific output.

**Interface:**
```bash
bin/ece-build --target web -o dist/ src/main.scm src/utils.scm
bin/ece-build --target cl -o dist/ src/main.scm src/utils.scm
```

Or with a response file for many inputs:
```bash
bin/ece-build --target web -o dist/ @files.txt
```

**Why shell, not ECE?** The script needs to invoke `qlot exec sbcl`, copy files, and generate directories — tasks where shell is natural. The compilation itself happens in ECE. The script is thin glue (~50-80 lines).

### 5. Web target reuses existing sandbox infrastructure

**Choice:** The web target copies `wasm/runtime.wasm`, `wasm/glue.js`, and bootstrap .ecec files, then wraps the user's .ecec bundle as a JS file (base64-encoded or as a string). A template `index.html` ties it all together.

**File structure:**
```
dist/
  index.html          ← from templates/web/index.html (app name substituted)
  ece-runtime.js      ← WASM binary (base64) + glue code (generated by build)
  ece-bootstrap.js    ← bootstrap .ecec files concatenated as JS data
  app.js              ← user .ecec bundle as JS data
```

This matches the existing sandbox structure. The `index.html` template includes `<script src>` tags for each JS file — the one loading mechanism browsers don't block on `file://`.

### 6. CL target is a self-contained directory

**Choice:** The CL target copies `src/runtime.lisp`, `bootstrap/*.ecec`, and the user's .ecec bundle into the output directory, plus a generated `run.lisp` that boots and loads the app.

**File structure:**
```
dist/
  runtime.lisp        ← copy of src/runtime.lisp
  bootstrap/           ← copy of bootstrap/*.ecec
  app.ecec            ← user .ecec bundle
  run.lisp            ← generated: (load "runtime.lisp") (load-bundle "app.ecec")
```

Run with: `sbcl --load dist/run.lisp`

## Risks / Trade-offs

**[Bundle file size]** A multi-space bundle is the sum of all individual .ecec files. For large apps this could be several MB. → Acceptable for the current use case. Future optimization (compression, binary format) can be added later.

**[File order is user-specified]** Users must list files in dependency order. A file that references a function defined in a later file will fail at runtime. → This matches `(load ...)` semantics. Users already understand load order. Error messages will make the cause clear.

**[Web packaging duplicates sandbox logic]** The `ece-build` web target partially duplicates `scripts/build-sandbox.sh`. → Refactor `build-sandbox.sh` to use `ece-build` internally, eliminating duplication.

**[No ASDF/qlot in CL target]** The CL target copies `runtime.lisp` directly. It doesn't use ASDF or qlot — the user runs `sbcl --load run.lisp`. → This is intentional: the output should be self-contained with no build tool dependencies.
